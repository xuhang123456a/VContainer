# M8 编译期源生成器 · 考题

## 🟢 概念题

1. 源生成器产出的注入器类叫什么命名约定？运行时谁、怎么找到它？
2. 生成的注入器和反射注入器实现的是同一个接口吗？
3. 源生成器运行在编译期还是运行时？它的产物进运行时程序集吗？
4. 哪些程序集会被源生成器跳过（不生成）？
5. 一个类型源生成器拒绝生成时，运行时会怎样？

<details><summary>参考答案要点</summary>

1. `{FullName}GeneratedInjector`。运行时 `InjectorCache.GetOrBuild` 用 `key.Assembly.GetType($"{key.FullName}GeneratedInjector")` 找到并 `Activator.CreateInstance`。
2. 是，都实现 `VContainer.IInjector`（CreateInstance + Inject）。
3. 编译期运行（Roslyn 生成器）；产物是 `.g.cs` 源码，被编译进用户程序集，所以**运行时存在的是编译产物**，生成器本身不进运行时。
4. VContainer 自身（非 Test）、UnityEngine.*/Unity.* 程序集、以及未引用 VContainer 的程序集。
5. 回退反射：`InjectorCache` 找不到生成类型 → `ReflectionInjector.Build`。
</details>

## 🟡 机制题

1. 生成的 `CreateInstance` 体内如何取构造参数？与反射版的数据流有何关系？
2. 增量管线的两条类型候选来源是什么？为什么要去重？
3. 源生成器对 Unity Component 的 `CreateInstance` 生成了什么？为什么？
4. 生成器的构造选择规则是什么？和运行时 `TypeAnalyzer` 一致吗？
5. `[Key]` 在生成代码里如何体现？
6. 哪些写法会触发诊断并导致不生成？

<details><summary>参考答案要点</summary>

1. 生成 `new T((P0)resolver.ResolveOrParameter(typeof(P0),"name",parameters,key), ...)`，调用的是与反射版**同一个** M4 扩展 `ResolveOrParameter`，数据流完全一致，只是反射版用 `ctor.Invoke` 而生成版用 `new`。
2. ① 显式 `[Inject]` 的 class 声明；② `Register*` 调用点反推的类型。`Concat` 后 `DistinctBy(Symbol)` 去重，避免同类型生成两次造成重复类名编译失败。
3. 生成 `throw new NotSupportedException("...cannot be new")`。Component 不能 `new`，只能对已存在实例 `Inject`。
4. 多 `[Inject]` 构造报错 → 唯一 `[Inject]` 构造 → 参数最多的公开构造 → 隐式无参构造 → 都没有报 ConstructorNotFound。与运行时 `TypeAnalyzer` 启发式一致（编译期/运行时对称）。
5. `ResolveOrParameter(typeof(T), "name", parameters, <keyValue>)` 的最后一个参数，由 `EmitKeyValue` 把 `[Key]` 的值转成字面量（字符串加引号、枚举转 `(EnumType)value`、null 为 "null"）。
6. 嵌套类型、抽象类、泛型类型、私有字段/属性/方法/构造、init-only 属性、多 `[Inject]` 构造、泛型方法、多个 `[Inject]` 方法等。
</details>

## 🔴 架构陷阱题

1. **核心不变量**：为什么生成器的构造选择、成员扫描、`ResolveOrParameter` 调用形态必须与运行时反射版逐条对称？不对称会出现什么"幽灵 bug"？
2. 一个项目里有的类型被生成、有的回退反射，这会造成运行时行为差异吗？为什么？
3. 如果命名约定 `{FullName}GeneratedInjector` 在生成器和 `InjectorCache` 两边不一致（比如一边加了下划线替换），会发生什么？
4. 生成器若不做 `DistinctBy(Symbol)` 去重，最坏后果是什么？是运行时错误还是编译错误？
5. 把一个 `private` 字段标 `[Inject]`，反射版能注入吗？源生成版呢？这暴露了两者什么差异？
6. 源生成器报了"泛型不支持"诊断，但用户仍用了泛型注入类型，运行期能正常工作吗？靠什么？
7. 为什么源生成器对"未生成的类型"要保证隐式回退，而不是直接编译失败？这体现什么设计哲学？

<details><summary>参考答案要点</summary>

1. 因为 `InjectorCache` 会在"有生成器"时优先用生成版、否则用反射版。若两者构造选择/参数顺序/key 取值不一致，就出现"装了源生成器包 → 注入结果 A；没装 → 注入结果 B"的环境相关 bug，极难定位（CI 与本地、Editor 与打包可能不同）。
2. 不应造成差异——这正是对称性要保证的。两者数据流都经 `ResolveOrParameter`，生命周期/缓存都在容器侧（M4），注入器只负责"取值+赋值"。只要对称，生成与反射结果一致。
3. 运行时 `InjectorCache` 按它那侧的名字找不到生成类型 → 全部回退反射。源生成器白干了（生成的类没人用），性能优化失效但功能仍正常（反射兜底）。
4. 同一类型生成两个同名 partial/class → **编译错误**（重复定义）。是编译期失败，比运行时错误更早暴露。
5. 反射版能（`FieldInfo.SetValue` 可设私有字段）；源生成版不能（生成的是 `__x.field=...` 直接赋值，访问不到 private）→ 报 `PrivateFieldNotSupported` 诊断、不生成、回退反射。这暴露"反射无视可见性、生成代码受 C# 可见性约束"的本质差异。
6. 能。该类型不生成注入器 → 运行时 `InjectorCache` miss → `ReflectionInjector` 用反射处理泛型类型。靠反射兜底。
7. 因为源生成是"性能优化"而非"功能前提"。它无法覆盖全部 C# 写法，若不支持就编译失败会让大量合法代码无法编译。隐式回退体现"渐进增强 / 优雅降级"——能优化就优化，不能就退回正确但慢的路径，功能永远可用。
</details>

## ✍️ 实操题

1. 用 Facade 的 `InjectorEmitter.TryEmit` 对一个含构造注入 + 字段注入 + 方法注入的类生成源码，肉眼核对生成的 `ResolveOrParameter` 调用与该类成员一一对应。
2. 给 Facade 增加"私有字段标 [Inject] 时返回 null + reason='private field'"的诊断，验证它会触发回退（reason 非空）。对照 Emitter 的 `CanBeCallFromInternal` 检查。
3. 写一段说明：给定同一个类，分别用 M2 的 `ReflectionInjector` 和 M8 生成的注入器构造实例，论证两者产出的对象在依赖注入结果上等价（指出依据是同一个 `ResolveOrParameter` + 同一容器缓存）。

<details><summary>参考答案要点</summary>

1. 生成的 CreateInstance 参数数 == 构造参数数；Inject 里字段赋值数 == [Inject] 字段数、方法调用数 == [Inject] 方法数，每处 `ResolveOrParameter` 的 typeof/参数名与成员对应。
2. 在 `EmitInject` 字段循环里加 `if (f.IsPrivate) { reason="private field"; return false; }`（需把 EmitInject 改为带 reason 返回），`TryEmit` 据此返回 null。验证 reason=="private field"。
3. 依据：两者注入器体内都调用同一个 `resolver.ResolveOrParameter(type,name,parameters,key)`；实例生命周期/单例缓存由容器（M4 `sharedInstances`）决定而非注入器；故相同注册下，相同依赖被解析为相同实例，最终对象状态等价。差异仅在构造方式（Invoke vs new）与性能。
</details>
