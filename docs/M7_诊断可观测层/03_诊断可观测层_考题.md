# M7 诊断可观测层 · 考题

## 🟢 概念题

1. 诊断采集默认是开启还是关闭？由什么控制？
2. `DiagnosticsCollector` 的采集器是按什么粒度组织的（每容器/每作用域/全局单例）？
3. `RegisterInfo` 里的 `StackTrace` 是用来做什么的？
4. `ResolveInfo` 记录了哪些统计量？
5. 关闭诊断时，容器解析的额外开销是多少？

<details><summary>参考答案要点</summary>

1. 默认关闭，由 `VContainerSettings.EnableDiagnostics`（`DiagnosticsEnabled`）控制；只有开启时 LifetimeScope 才注入 Collector。
2. 每作用域一个 Collector，全局 `DiagnositcsContext` 用 scopeName 字符串作键管理所有 Collector。
3. 抓取注册代码的调用栈，定位到"这条注册写在哪个文件哪一行"，供编辑器点击跳转。
4. Registration、Instances 列表、MaxDepth、RefCount、ResolveTime。
5. 近乎为零——所有采集点是 `Diagnostics?.Xxx`，null 时 `?.` 短路，`Resolve` 直接走 `ResolveCore`。
</details>

## 🟡 机制题

1. `TraceResolve` 如何在不改变解析逻辑的情况下统计耗时？
2. 解析依赖树是如何被运行时重建的？用了什么数据结构？
3. Transient 与 Singleton 的 `ResolveTime` 聚合方式为何不同？
4. `TraceBuild` 解决了什么"声明态到构建态"的衔接问题？
5. 为什么 `TraceResolve` 要跳过 `CollectionInstanceProvider` 和 `current == owner` 的情况？
6. `GetGroupedDiagnosticsInfos` 为什么过滤 `MaxDepth <= 1`？

<details><summary>参考答案要点</summary>

1. 把真实解析逻辑 `ResolveCore` 作为 `Func<Registration,object>` 委托传入，采集器在调用它的前后包 `Stopwatch.Start/Stop`，逻辑本身不变。
2. `ThreadLocal<Stack<DiagnosticsInfo>>` 维护当前解析链：进入解析时把当前节点 Add 到栈顶 owner 的 Dependencies 并压栈、解析完弹栈，递归过程就地织出依赖树。
3. Transient 每次新建（多次解析），用移动平均反映平均构造成本；Singleton/Scoped 只构造一次，取最大值反映那次（含依赖）的真实耗时。
4. 注册时只有 RegistrationBuilder（TraceRegister 记一条），构建后才有 Registration；TraceBuild 按 `RegisterInfo.RegistrationBuilder == builder` 找回那条并回填 ResolveInfo，把两态关联。
5. 集合解析会对每个成员再解析，记录集合本身会产生噪声/重复；`current==owner` 是自引用（同一节点解析自己），避免错误的自依赖边。
6. 顶层注册才作为列表项展示，深度 >1 的是别人的依赖，已作为子节点嵌套在 Dependencies 树里，避免重复平铺。
</details>

## 🔴 架构陷阱题

1. **核心不变量**：若在 `TraceResolve` 的"开启分支"里顺手加了实例缓存而"关闭分支"没有，会违反什么原则？会导致什么难查的 bug？
2. 诊断开启 vs 关闭，单例的"构造次数"会变吗？为什么这点必须保证？
3. `ThreadLocal` 调用栈若改成普通字段（非线程本地），在什么场景出错？
4. `RegisterInfo` 的 StackTrace 抓取在 IL2CPP/Release 构建下可能拿不到文件名，框架如何容错？
5. 一个对象被解析 3 次（Transient），`ResolveInfo.Instances` 里有几个对象？`RefCount` 是多少？两者为何可能不一致？
6. 把采集点从 `Diagnostics?.TraceResolve(...)` 改成无条件 `TraceResolve`（内部判 null）有什么坏处？
7. `RemoveCollector` 若忘了在作用域销毁时调用，会有什么后果？

<details><summary>参考答案要点</summary>

1. 违反"观测不改变被观测对象行为"。会导致"开诊断时正常、关诊断时 bug"（或反之）的海森堡 bug，极难复现定位。
2. 不会变。开启只是多了计时/记录，`resolving(registration)` 仍是同一个 `ResolveCore`，单例仍走 `Lazy` 构造一次。必须保证，否则诊断会篡改生命周期语义。
3. 多线程并发解析时，多个线程共享同一个解析栈会互相污染依赖树（A 线程的解析被记成 B 线程的依赖），且压/弹栈竞态导致栈错乱。ThreadLocal 隔离每线程独立栈。
4. `GetFilename/GetFileLineNumber` 捕获 `NotSupportedException`/`SecurityException`，捕获后置 `displayFileNames=false` 停止再尝试，`GetHeadline` 降级为只显示类型+方法名（或 IL offset）。
5. Transient 每次新建 → Instances 有 3 个不同对象、RefCount=3。一致。但若是 Singleton 解析 3 次：RefCount=3 但 Instances 去重后只有 1 个（`!Contains` 守卫）——此时两者不一致，正反映"被引用 3 次但只有 1 个实例"。
6. 即便关闭也要进一次方法调用 + 内部 null 判断 + 可能的参数求值（如 `new RegisterInfo(...)` 提前构造抓栈），破坏"关闭零成本"。`?.` 在外层短路能避免连参数都不构造。
7. Collector 泄漏在全局 `collectors` 字典里不被回收，scopeName 相同的新作用域可能复用旧采集器的脏数据，且内存随场景切换累积增长。
</details>

## ✍️ 实操题

1. 用 Facade 写测试：构造 A 依赖 B 的解析链，断言解析 A 后 `Find(A).Dependencies` 含 B，且 B 的 `MaxDepth` 比 A 大。
2. 验证"零成本不变量"：同一组注册，分别在 `Diagnostics=null` 与非 null 下解析单例，断言两种情况下返回的实例引用相同（行为等价）。
3. 给 Facade 增加"按 scopeName 分组 + 过滤顶层（MaxDepth<=1）"的 `GetGrouped` 视图，对照 `GetGroupedDiagnosticsInfos`。

<details><summary>参考答案要点</summary>

1. 解析 A 时其构造内解析 B（嵌套 TraceResolve），断言 `Find(regA).Dependencies.Contains(Find(regB))` 且 `Find(regB).ResolveInfo.MaxDepth > Find(regA).ResolveInfo.MaxDepth`（B 在更深栈层）。
2. 两次 `Resolve(singletonReg)`：`var x1 = withDiag.Resolve(); var x2 = withoutDiag.Resolve();`（同容器或等价容器），断言单例语义一致——关键是诊断分支只透传委托，不改 ResolveCore。
3. 给每条 info 附 scopeName，`GetGrouped` 用 LINQ `Where(MaxDepth<=1).ToLookup(scopeName)`。
</details>
