# M2 类型分析与反射注入器 · 考题

## 🟢 概念题

1. `IInjector` 的两个方法 `Inject` 与 `CreateInstance` 分别在什么场景被调用？
2. 当一个类型没有任何 `[Inject]` 构造时，`TypeAnalyzer` 如何选择构造函数？
3. `InjectorCache.GetOrBuild` 选择注入器的三个优先级是什么？
4. `[Key]` 特性是在解析时读取还是分析时读取？
5. 反射注入器的成员注入顺序是怎样的？

<details><summary>参考答案要点</summary>

1. `CreateInstance` 用于「容器创建新实例」（构造 + 注入）；`Inject` 用于「对已存在实例注入」（如 `container.Inject(monoBehaviour)`，MonoBehaviour 由 Unity new 出来不能由容器构造）。
2. 选参数数量最多的构造函数。
3. ① 同程序集的 `{FullName}GeneratedInjector`（源生成）→ ② IL 织入静态方法 `__GetGeneratedInjector`（已废弃）→ ③ `ReflectionInjector`。
4. 分析时（构造 `InjectXxxInfo` 时一次性抽取到 `ParameterKeys`/`Key`），解析时不再反射读特性。
5. 字段 → 属性 → 方法（构造注入更早，在 `CreateInstance` 内、`Inject` 之前）。
</details>

## 🟡 机制题

1. 同名字段在子类和父类都被 `[Inject]` 标注，会发生什么？方法 override 呢？
2. `ResolveOrParameter` 在 `parameters` 非空时的完整匹配逻辑是什么？
3. 为什么循环依赖检测能豁免 `FuncInstanceProvider` 提供的依赖？
4. `TypeAnalyzer.AnalyzeWithCache` 与 `InjectorCache.GetOrBuild` 各缓存什么？两者关系？
5. `CreateInstance` 解析参数失败时，异常如何被加工？
6. 循环依赖检测为什么用 `[ThreadStatic]` 的 `Stack<DependencyInfo>`？

<details><summary>参考答案要点</summary>

1. 字段按 `Name` 去重，子类同名只取一次（注意：同一类内重复同名字段会抛异常）；方法按 `GetBaseDefinition()` 去重，override 不会被重复 invoke。
2. 遍历 `parameters`，第一个 `Match(type,name)` 为真的覆盖参数直接 `GetValue` 返回；都不命中则按 key（或无 key）向容器 `Resolve`。
3. 因为 `Func<T>` 是延迟工厂，用户可在闭包里手动决定何时/是否取依赖，抽象环可由用户避免，故检测器对其 `return` 跳过。
4. `TypeAnalyzer` 缓存 `InjectTypeInfo`（注入计划）；`InjectorCache` 缓存 `IInjector` 实例。`ReflectionInjector.Build` 内部会调 `AnalyzeWithCache`，因此 injector 持有 typeInfo 引用。
5. 捕获 `VContainerException`，重新包装为 `Failed to resolve {injectTypeInfo.Type} : {原消息}`，保留 `InvalidType`，形成可读的解析链。
6. 检测在构建期可能并行（每线程独立 DFS 栈），且要复用避免每个注册都分配新栈——ThreadStatic 兼顾线程安全与零额外分配。
</details>

## 🔴 架构陷阱题

1. **反射版 vs 源生成版**：两者都实现 `IInjector`，在「数据流」上有何相同、在「成本/时机」上有何不同？哪些类型源生成器会拒绝生成（回退反射）？
2. 漏掉「方法 override 去重」会导致什么具体后果？
3. 如果在 `CreateInstance` 里把 `Rent` 的参数数组在 `Invoke` 之前就 `Return`，会发生什么？
4. 一个 MonoBehaviour 类型走 `CreateInstance` 会怎样？为什么 VContainer 对 Unity Component 特殊处理？
5. 覆盖参数同时有一个 `TypedParameter(typeof(int))` 和一个 `NamedParameter("count")`，构造参数 `int count` 会用哪个？换顺序结果会变吗？
6. 循环依赖检测发生在「构建期」而非「解析期」，这带来什么好处和什么盲区？
7. `[Inject]` 标注了两个构造函数会怎样？只标注一个但它参数比另一个未标注的少呢？

<details><summary>参考答案要点</summary>

1. 数据流相同：都经 `resolver.ResolveOrParameter(type,name,parameters,key)` 取依赖、构造、注入。不同：反射版运行时用 `MethodInfo.Invoke`（有装箱/反射开销 + 参数数组租借），源生成版是编译期直接 `new T(...)` + 强类型赋值（无反射）。源生成器拒绝：嵌套类型、抽象类、泛型类型、私有成员/构造、多 `[Inject]` 构造、泛型方法等（见 M8 Emitter 的诊断分支），这些回退反射。
2. 同一个 `[Inject]` 方法（基类定义、子类 override）会被 invoke 两次，造成重复副作用（如重复订阅事件、重复初始化）。
3. 数组被 `Array.Clear` 清空，`Invoke` 拿到的参数全变 null/default，构造出错误对象或 NRE。租借必须包到 `Invoke` 之后、`finally` 里归还。
4. `ReflectionInjector.CreateInstance` 会尝试 `ctor.Invoke`；但 Component 不能 `new`（Unity 限制）。VContainer 的做法是**不让 Component 走 CreateInstance**——它通过 `RegisterComponent*`/`InjectGameObject` 走 `Inject`（对已存在实例注入）。源生成器更是直接对 Component 生成 `throw NotSupportedException`。`TypeAnalyzer` 也对 Component 放宽"无构造"限制（Unity 可能 strip 构造）。
5. 按 `Parameters` 列表中先添加的先匹配。`int count`：`TypedParameter` 用 `type==int` 匹配、`NamedParameter` 用 `name=="count"` 匹配，两者都会命中——谁在列表里靠前用谁。换顺序结果会变，这是个隐蔽陷阱。
6. 好处：环在容器构建时（启动期）即抛出并打印完整路径，不必等到首次解析才崩、定位清晰。盲区：只能检测「分析过 + 在 Registry 能 TryGet 到」的静态依赖边；通过 `Func<T>`/运行时动态 `Resolve` 形成的环检测不到（也正因此 Func 被豁免）。
7. 两个 `[Inject]` 构造 → 抛 `multiple [Inject] marked constructors`。只标一个即使参数更少，也强制选它（`[Inject]` 优先级高于"参数最多"启发式）。
</details>

## ✍️ 实操题

1. 给 Facade 的 `ReflectionInjector` 补上「属性注入」，注意它要在字段之后、方法之前执行，并复用同名去重逻辑。
2. 实现一个最小循环依赖检测：输入若干 `(Type, 依赖Type[])`，用 DFS + 栈检测环并打印路径，且对标记为「工厂」的依赖跳过。对照 `TypeAnalyzer.CheckCircularDependencyRecursive` 验证思路一致。
3. 写一个测试证明「`[Inject]` 构造优先级高于参数最多构造」：一个类有 `Ctor(a,b)` 无标注、`Ctor(a)` 标注 `[Inject]`，断言容器选了后者。

<details><summary>参考答案要点</summary>

1. 仿照字段循环，扫描 `t.GetProperties(flags)` 取 `[Inject]` 且可写的属性，按 `Name` 去重，`SetValue`。插入到字段循环之后、方法循环之前。
2. DFS：进入节点先检查是否已在 `stack`（按 Type 比较）→ 在则报环（`stack.Take(i+1).Reverse()` 拼路径）；否则压栈、递归每个依赖（工厂依赖 `continue` 跳过）、回溯弹栈。
3. 注册该类，断言构造出的实例是经 `Ctor(a)` 路径（如让两个构造记录不同标记字段，检查标记值）。预期选 `[Inject]` 标注的单参构造。
</details>
