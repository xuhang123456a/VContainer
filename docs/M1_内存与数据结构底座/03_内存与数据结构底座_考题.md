# M1 内存复用与数据结构底座 · 考题

## 🟢 概念题

1. `CappedArrayPool.Rent(5)` 返回的数组长度一定是多少？与 `System.Buffers.ArrayPool` 的关键语义差异是什么？
2. `FreeList` 删除一个中间元素时，后面的元素索引会变吗？为什么这点重要？
3. `ListPool<T>.Get(out var buf)` 返回的 `BufferScope` 起什么作用？
4. `FixedTypeObjectKeyHashtable` 构建完成后还能不能写入新键值？
5. `CompositeDisposable` 的释放顺序是 FIFO 还是 LIFO？

<details><summary>参考答案要点</summary>

1. 恰好 5。`ArrayPool` 返回 **≥** 请求长度的数组，而 `CappedArrayPool` 按精确长度分桶、返回**恰好**等长数组，因为它服务于 `MethodInfo.Invoke(args)`，参数数组长度必须精确。
2. 不变。`RemoveAt` 只把该槽置 null，不搬移后续元素 → 索引稳定，使得遍历期间删除安全。
3. 它是个 `IDisposable` struct，`using` 块结束时自动把 List `Clear` 后压回池，避免忘记 `Release`。
4. 不能。它只有构造期一次性填充 + `TryGet`，无写 API，是"冻结只读"表。
5. LIFO（`Stack` 弹栈），后注册的先释放——符合依赖释放顺序。
</details>

## 🟡 机制题

1. 桶满（`tail >= bucket.Length`）时 `CappedArrayPool` 如何扩容？旧数组怎么处理？
2. `FreeList.Add` 找空位的策略是什么？满了按什么倍率扩容？
3. `FixedTypeObjectKeyHashtable` 为什么用 `RuntimeHelpers.GetHashCode(type)` 而不是 `type.GetHashCode()`？
4. `FreeList` 的 `lastIndex` 字段在删除尾部元素时如何更新？
5. `CappedArrayPool.Return` 如何知道数组属于哪个桶？它会校验数组是不是自己借出去的吗？
6. `Shared8Limit` 的 8 是什么含义？请求长度 9 会发生什么？

<details><summary>参考答案要点</summary>

1. `Array.Resize(ref bucket, bucket.Length*2)` 把存放数组指针的桶翻倍，旧的数组引用被复制进新桶继续保留复用。
2. 优先复用 null 空洞（`FindNullIndex`）；满了按 `len + len/2`（×1.5）扩容。
3. 绕过用户类型可能重写的 `GetHashCode`，取得稳定且廉价的引用哈希，保证容器内 Type 键的确定性。
4. 若删的就是 `lastIndex`，从该位置往前回退扫描到第一个非 null 索引（`FindLastNonNullIndex`）；不缩容数组。
5. 仅按 `array.Length-1` 定位桶。**不校验归属**，只 `Array.Clear` + `tail--`，借出计数模型而非归属追踪。
6. maxLength=8，即支持长度 1~8 的数组入池。长度 9 走 `length > buckets.Length` 分支直接 `new T[9]`，不入池也不复用。
</details>

## 🔴 架构陷阱题

1. **同底座对比**：`CappedArrayPool` 和 `ListPool` 都是对象池，它们在「容量模型」和「归还时机」上有什么本质不同？
2. 如果把 `Rent` 来的参数数组保存到某个长生命周期字段而不立刻 `Return`，会出现什么具体故障？故障表现是什么？
3. `FreeList` 若改成「删除时把末尾元素搬到被删位置（swap-remove）」来避免空洞，会破坏哪个上层（M6）依赖的不变量？
4. 漏掉 `try/finally` 中的 `Return`（异常路径忘还）会导致什么后果？是内存泄漏还是别的？
5. `FixedTypeObjectKeyHashtable` 用「整桶 Array.Copy 扩容」做拉链，当某个哈希桶冲突很多时会有什么性能特征？
6. 为什么 `TypeKeyHashTable2` 这种更先进的 Robin Hood 表存在却没被 `Registry` 使用？把它强行换上去要注意什么键差异？
7. 多线程同时 `Rent`/`Return` 同一个 `Shared8Limit`，锁粒度是怎样的？会成为并行容器构建的瓶颈吗？

<details><summary>参考答案要点</summary>

1. `CappedArrayPool` 用「每长度一个栈顶指针」的借出计数模型、按精确长度分桶；`ListPool` 用「单一 Stack 复用任意容量 List」。前者归还按长度找桶清引用，后者归还 `Clear` 后压栈。前者数组长度精确，后者 List 容量任意（≥某默认）。
2. 该数组随后可能被另一次 `Rent` 借出（计数模型不知它仍被持有），或在 `Return` 时被 `Array.Clear` 清空。表现为「字段里的数据莫名变 null / 被别处内容覆盖」，是典型的 use-after-return。
3. 破坏「索引稳定」不变量。`PlayerLoopRunner.Run` 用固定 `span.Length` 按 index 遍历，swap-remove 会把还没遍历的末尾元素挪到已遍历位置，导致**漏 tick**或**重复 tick**。
4. 借出计数 `tail` 不回退，桶里的可复用槽逐渐"看起来被占满"，迫使 `Rent` 频繁触发 `Array.Resize` 或新建数组——是**复用退化/隐性 GC 升高**，不是经典内存泄漏（数组仍可被 GC），但池效果丧失。
5. 每次插入冲突要 `new + Array.Copy` 整桶；查表是桶内线性扫描。冲突集中时退化为 O(链长) 且构建期分配增多。但因表是构建期一次性建好、运行期只读，热路径仍可接受。
6. 它是备选/演进中的实现，当前 `Registry` 只接受 `(Type,object)` 复合键，而 `TypeKeyHashTable2` 键是单 `Type`，不支持 Keyed 注册。强行替换需把键扩展为 `(Type,object)` 或额外维护 key 维度，否则 Keyed 服务会全部哈希冲撞到同一 Type 槽。
7. 单一 `syncRoot` 全表锁，`Rent` 与 `Return` 全程持锁。在 `VCONTAINER_PARALLEL_CONTAINER_BUILD` 并行建表时，注入参数数组的租借会竞争该锁，但租借区间极短（填几个槽），实测瓶颈通常在反射本身而非此锁。（"通常不是瓶颈"为推断，未在本仓库基准验证。）
</details>

## ✍️ 实操题

1. 给 `CappedArrayPool` 增加一个 `RentExact(int)` 的调试版本：在 `Return` 时校验归还的数组确实是本池借出的（用 `HashSet<T[]>` 追踪）。说明这会带来什么开销，以及为什么生产版本不这么做。
2. 用本模块的 `FreeList` 写一个「每帧执行、可在执行中注销自己」的极简调度器，并构造一个测试证明「注销当前项不会漏执行后续项」。
3. 把 `FixedTypeObjectKeyHashtable` 改造成支持「同键覆盖」（后注册覆盖先注册），需要改哪一处？这会破坏"只读"语义吗？请权衡。

<details><summary>参考答案要点</summary>

1. 用 `HashSet<T[]>` 记录借出引用，`Return` 时 `Contains` 校验。开销：每次 Rent/Return 多一次哈希集合增删 + 集合自身内存；破坏无锁/低开销目标。生产不做是因为热路径调用极频繁，且 `try/finally` 配对已由调用方保证。
2. 见 M1-02 的 FreeList 使用示例：`Run` 遍历 `[0,Length)`、跳过 null、执行后 `RemoveAt(i)`。测试：注册 3 个项，第 2 个在执行时注销自己，断言第 3 个本帧仍被执行（因为遍历按 index 且只 null 化）。
3. 在构造时的拉链插入处，遇到相同 `(Type,key)` 改为覆盖而非追加。它会破坏"构建后不可变"的纯只读语义（变成"构建期可覆盖"），但因仍在构造函数内、对外仍只暴露 `TryGet`，运行期只读不变——可接受。实际 VContainer 的覆盖逻辑在上层 `Registry.AddToBuildBuffer` 用 Dictionary 完成，而非在哈希表内（见 M3）。
</details>
