# M4 容器解析与作用域树 · 考题

## 🟢 概念题

1. `Transient` / `Singleton` / `Scoped` 三种生命周期在"实例缓存"上分别如何处理？
2. `Container` 内部为什么还持有一个 `rootScope`(ScopedContainer)？
3. 单例实例缓存的字典键是什么类型？为什么能这样？
4. 哪种生命周期的实例**不会**被容器登记进 Dispose 跟踪？
5. `Resolve(Type)` 和 `Resolve(Registration)` 的查找行为有何不同？

<details><summary>参考答案要点</summary>

1. Transient 每次新建不缓存；Singleton 缓存在"声明它的作用域层"的 `sharedInstances`；Scoped 缓存在当前作用域的 `sharedInstances`。
2. 为了统一语义——Container 把 Scoped 解析委托给 rootScope，自己只管 Singleton；rootScope 是 parent=null 的 ScopedContainer，复用同一套作用域逻辑。
3. `Registration` 引用。因为 Registration 不可变、无实例状态，可作稳定 key 在父子作用域共享、命中同一缓存条目。
4. Transient（不跟踪）；以及由 `ExistingInstanceProvider` 提供的实例（RegisterInstance，外部拥有）。
5. `Resolve(Type)` 沿作用域链 `TryFindRegistration` 查找；`Resolve(Registration)` 直接对已知注册解析，不查找。
</details>

## 🟡 机制题

1. `Lazy<object>` 在这里解决什么并发问题？
2. `CreateTrackedInstance` 里 `created = lazy.IsValueCreated` 为什么要在 `lazy.Value` 之前读？
3. `ScopedContainer.ResolveCore` 解析 Singleton 的三个分支判断分别是什么？
4. `createInstance` 为什么做成字段委托而不是每次 new 闭包？
5. 容器 `Dispose` 时对子作用域会怎样？子作用域需要自己 Dispose 吗？
6. `IObjectResolver` 是怎么变成"可注入的当前容器"的？

<details><summary>参考答案要点</summary>

1. 并发解析同一单例时，`ConcurrentDictionary.GetOrAdd` + `Lazy`(默认 ExecutionAndPublication 模式) 保证工厂只执行一次、所有线程拿到同一实例。
2. 取 `Value` 会触发构造并把 `IsValueCreated` 变 true。若取值后读，永远 true，导致"本次首次创建"判不出来，可弃实例漏登记 Dispose。
3. ① `Parent is null`(根作用域)→ `Root.Resolve`；② 本作用域 registry `Exists` 该 impl → 本层 `CreateTrackedInstance`；③ 否则 `Parent.Resolve` 向上找拥有者。
4. 避免每次 `GetOrAdd` 都分配一个新的工厂委托闭包（GetOrAdd 的 valueFactory 参数）；缓存成字段复用，减少 GC。
5. 不级联。父 `Dispose` 只释放自己 `disposables` 里登记的实例，不自动 Dispose 子作用域。子作用域（如 LifetimeScope）需在自身 OnDestroy 时各自 Dispose。
6. `BuildRegistry` 末尾追加 `typeof(IObjectResolver)` Transient 注册，Provider 是 `ContainerInstanceProvider`，其 `SpawnInstance(resolver) => resolver` 返回当前传入的 resolver。
</details>

## 🔴 架构陷阱题

1. **同底座对比**：`Container` 与 `ScopedContainer` 都有 `sharedInstances` + `disposables`，但 Singleton 解析逻辑不同，差异点是什么？为什么根容器不需要"向上路由"？
2. 父作用域注册了单例 `IFoo`，子作用域**也**注册了单例 `IFoo`，在子作用域 `Resolve<IFoo>()` 得到的是父的还是子的实例？依据是哪行逻辑？
3. 在子作用域解析一个"只在根注册的 Singleton"，实例缓存在哪个作用域的 `sharedInstances`？两个不同子作用域解析它会得到同一个吗？
4. 如果把 Dispose 守卫里的 `!(reg.Provider is ExistingInstanceProvider)` 去掉，会发生什么用户可感知的 bug？
5. 一个 Scoped 服务在根容器（非作用域）被解析会怎样？
6. 容器 Dispose 的释放顺序若改成 FIFO，可能引发什么问题？
7. 为什么"运行时不能新增注册，只能 CreateScope"？这与 M3 的哪个设计直接相关？

<details><summary>参考答案要点</summary>

1. `ScopedContainer` 的 Singleton 要判断"该单例归属哪层"并可能向上路由；`Container`(根) 是单例的最终拥有者，直接本地 `GetOrAdd` 即可，无更上层可路由。根的 parent 概念上为 null。
2. 子的。子作用域 registry `Exists(IFoo)` 为真 → 走"本层 CreateTrackedInstance"分支，在子作用域建并缓存，不上抛父。这实现了子作用域对单例的"遮蔽/覆盖"。
3. 缓存在**根**的 `sharedInstances`（向上路由到 `Root.Resolve`，根 GetOrAdd）。两个不同子作用域解析它得到同一个（都路由到根、命中同一 Registration 键）。
4. `RegisterInstance` 注册的外部对象会被容器在 Dispose 时调用其 `Dispose()`，可能把"用户还在用、生命周期不归容器管"的对象提前释放，导致 ObjectDisposed 异常或资源被错误关闭。
5. 根 `Container.ResolveCore` 的 Scoped 分支委托给 `rootScope.Resolve`，在 rootScope（parent=null 的作用域）缓存——相当于把 Scoped 当作"根作用域级单例"处理，能解析但语义上等同根级缓存。
6. 后创建的对象往往依赖先创建的对象（如 service 依赖 logger）。FIFO 会先释放 logger，后释放 service 时 service.Dispose 里若用 logger 就访问到已释放对象。LIFO 保证依赖者先于被依赖者释放。
7. Registry 构建后基于 `FixedTypeObjectKeyHashtable` 冻结只读（M3），无写 API。新增需求只能通过 `CreateScope` 建带新 registry 的子作用域来满足。
</details>

## ✍️ 实操题

1. 给 Facade `Container` 增加 `TryResolve(Type, out object, key)` 与泛型 `Resolve<T>()` 门面，复用现有 `TryFindRegistration`。
2. 写测试验证三条单例路由：① 根注册单例两次解析同对象；② 子作用域解析根单例得到根的对象；③ 子作用域重注册同接口单例则遮蔽父。
3. 构造一个"Dispose 顺序"测试：注册 A(依赖 B) 与 B 两个可弃单例，解析 A，断言容器 Dispose 时 A 先于 B 被 Dispose。

<details><summary>参考答案要点</summary>

1. `TryResolve`：`TryFindRegistration` 成功则 `Resolve(reg)` 赋值返回 true，否则 default+false。`Resolve<T>()`：`(T)Resolve(typeof(T))`。
2. ① `root.Resolve(IFoo)==root.Resolve(IFoo)`；② `child.Resolve(IFoo)==root.Resolve(IFoo)`（向上路由）；③ child 重注册后 `child.Resolve(IFoo)!=root.Resolve(IFoo)`（本层遮蔽）。
3. A 解析时先构造 B（依赖）再构造 A，登记顺序 B 先入栈、A 后入栈；LIFO Dispose 弹栈先弹 A 后弹 B，故 A 先 Dispose。用记录 Dispose 时间戳/顺序列表断言。
</details>
